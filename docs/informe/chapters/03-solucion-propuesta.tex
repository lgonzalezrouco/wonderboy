\section{Solución Propuesta}
\label{sec:solucion-propuesta}

\subsection{Visión general}
\label{subsec:solucion-vision}

Los problemas del capítulo anterior se abordan con una separación estricta de
responsabilidades. La simulación de un frame vive en un núcleo puro que transforma un
mundo en otro mundo. La orquestación de la partida (vidas, fase de juego, puntaje entre
etapas) se expresa con una pila monádica sin \texttt{IO}. Los efectos del entorno (entrada,
reloj, archivos, dibujado) quedan en adaptadores y en el ensamblado del bucle Gloss.

\subsection{Núcleo de simulación}
\label{subsec:solucion-nucleo}

El agregado central es el mundo. Cada frame lo reemplaza por completo
(sección~\ref{subsubsec:problema-estado}). La denotación del paso en juego:

\begin{lstlisting}
advanceSimulationFrame
  :: FrameParams -> LevelCount -> DeltaTime -> Input
  -> PlayingFrame -> FrameResult

data PlayingFrame = PlayingFrame
  { pfWorld     :: World
  , pfLives     :: Lives
  , pfScore     :: Score
  , pfLevelIndex :: Int
  }

data FrameResult = FrameResult
  { frWorld  :: World
  , frLives  :: Lives
  , frScore  :: Score
  , frPhase  :: GamePhase
  }
\end{lstlisting}

Composición interna (orden fijo):

\begin{lstlisting}
wAfterFrame       = advanceFrame params life dt input (pfWorld playing)
wAfterCombat      = resolveCombat combat input wAfterFrame
wAfterProjectiles = resolveProjectiles throwP combat params dt input wAfterCombat
wAfterHazards     = resolveFallingHazards life combat dt wAfterProjectiles
wAfterBoss        = resolveBossPhases combat wBefore wAfterHazards
-- ... pickups, victoria hibrida, muerte, fase final
\end{lstlisting}

\texttt{advanceFrame} antecede a combate y reglas de victoria
(sección~\ref{subsec:pipeline-frame}). \texttt{updateGame} solo invoca
\texttt{advanceSimulationFrame} cuando la fase y el intervalo de tiempo lo permiten.

\subsection{Orquestación con mónadas}
\label{subsec:solucion-monadas}

La partida agrega a \texttt{GameState} lo que el mundo no cubre (vidas, fase,
puntaje del nivel, índice de etapa). \texttt{GameConfig} se sirve de solo lectura.
La pila:

\begin{lstlisting}
newtype GameM a = GameM {
  unGameM :: ReaderT GameConfig
             (StateT GameState (ExceptT GameError Identity)) a
}

updateGame :: (MonadReader GameConfig m, MonadState GameState m)
             => DeltaTime -> Input -> m ()
\end{lstlisting}

\texttt{ReaderT} provee configuración y \texttt{StateT} el estado de partida.
\texttt{ExceptT} errores de carga (\texttt{runGameM :: ... -> Either GameError (a, GameState)}).
\texttt{updateGame} es polimórfica en \texttt{MonadReader} y \texttt{MonadState} para
facilitar tests sin \texttt{GameM} concreto.

\subsection{Comportamiento e interpretación}
\label{subsec:solucion-behaviour}

Cada enemigo lleva un \texttt{BehaviourProgram} (\texttt{Free EntityAction}).
\texttt{runBehaviourStep} consume una instrucción por frame antes de \texttt{step}
(sección~\ref{subsec:dsl}).

\begin{lstlisting}
patrolHorizontal speed leg =
  setVelocity left
    >>> waitFrames leg
    >>> setVelocity right
    >>> waitFrames leg
    >>> patrolHorizontal speed leg
\end{lstlisting}

\paragraph{Mónada libre frente a ADT recursivo.}
Un ADT \texttt{data Behaviour = ... | Seq Behaviour Behaviour} con intérprete
también separa descripción de ejecución. Elegimos \texttt{Free EntityAction} para
reutilizar \texttt{Monad} (\lstinline|>>>|, \lstinline|fix|) y extender primitivas en el funtor
sin reescribir la forma recursiva.

\begin{lstlisting}
data EntityAction next
  = SetVelocity Velocity next
  | WaitFrames Frames next
  | IfPlayerWithinRange Float BehaviourProgram BehaviourProgram next
  | MoveTowardPlayer Float next
  | Shoot next
  deriving (Functor)

newtype BehaviourProgram = BehaviourProgram
  { unBehaviourProgram :: Free EntityAction () }
\end{lstlisting}
